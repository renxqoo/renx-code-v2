## Superpowers 工作流技能

源自 [obra/superpowers](https://github.com/obra/superpowers) 项目，是一套完整的 AI 辅助开发工作流框架。

### brainstorming - 头脑风暴

**使用场景：** 在任何创意工作之前必须使用——创建功能、构建组件、添加功能或修改行为。

**核心流程：**

1. 探索项目上下文
2. 提供视觉伴侣（如涉及 UI 问题）
3. 逐一提问澄清需求
4. 提出 2-3 个方案并推荐
5. 展示设计方案，获得用户批准
6. 编写设计文档并提交
7. 规范审查循环
8. 用户审查书面规范
9. 调用 `writing-plans` 创建实施计划

**铁律：** 在获得用户批准设计方案之前，禁止编写任何代码。

---

### writing-plans - 编写计划

**使用场景：** 有规格说明或需求，需要在接触代码之前创建实施计划。

**核心要求：**

- 假设工程师对代码库零上下文
- 提供完整信息：文件路径、代码、测试命令
- 分解为小型任务（每个步骤 2-5 分钟）
- 遵循 DRY、YAGNI、TDD、频繁提交原则

**计划结构：**

- 文件结构规划
- 每个任务包含：失败的测试 → 最小实现 → 验证通过 → 提交

---

### subagent-driven-development - 子代理驱动开发

**使用场景：** 在当前会话中执行具有独立任务的实施计划。

**核心优势：**

- 同一会话，无需上下文切换
- 每个任务派发新子代理，避免上下文污染
- 两阶段审查：规范合规性 → 代码质量
- 快速迭代

**执行流程：**

```
读取计划 → 派发实现子代理 → 规范审查 → 代码质量审查 → 下一任务
```

---

### executing-plans - 执行计划

**使用场景：** 有书面实施计划，需要在单独会话中执行并带有审查检查点。

**注意：** 建议在支持子代理的平台使用 `subagent-driven-development` 替代。

---

### using-git-worktrees - 使用 Git Worktrees

**使用场景：** 开始需要与当前工作空间隔离的功能工作，或执行实施计划之前。

**核心流程：**

1. 智能目录选择（`.worktrees` > CLAUDE.md 配置 > 询问用户）
2. 安全验证（确认目录被 .gitignore 忽略）
3. 创建 worktree
4. 运行项目设置（npm install / cargo build 等）
5. 验证干净基线（测试通过）

---

### finishing-a-development-branch - 完成开发分支

**使用场景：** 实现完成、测试通过后，决定如何集成工作。

**四个选项：**

1. **合并到主分支** - 本地合并并删除分支
2. **创建 PR** - 推送并创建 Pull Request
3. **保持分支** - 保留待后续处理
4. **丢弃** - 永久删除（需输入 "discard" 确认）

**铁律：** 必须验证测试通过后才能展示选项。

---

### test-driven-development - 测试驱动开发

**使用场景：** 实施任何功能或 bug 修复时，在编写实现代码之前。

**铁律：** 没有失败的测试就不能写生产代码。

**红-绿-重构循环：**

```
RED    → 编写失败的测试（验证失败原因正确）
GREEN  → 编写最小代码使测试通过
REFACTOR → 清理代码，保持测试绿色
```

**禁止行为：**

- 先写代码后补测试
- 跳过验证测试失败的步骤
- 测试通过后立即声称完成

---

### systematic-debugging - 系统调试

**使用场景：** 遇到任何 bug、测试失败或意外行为时，在提出修复方案之前。

**铁律：** 在找到根本原因之前禁止尝试修复。

**四阶段流程：**

| 阶段            | 活动                                             |
| --------------- | ------------------------------------------------ |
| 1. 根本原因调查 | 仔细阅读错误、复现问题、检查最近更改、追踪数据流 |
| 2. 模式分析     | 找到工作示例、比较差异、理解依赖                 |
| 3. 假设和测试   | 形成单一假设、最小化测试变量                     |
| 4. 实施         | 创建失败测试、实施单一修复、验证                 |

**警告：** 如果 3+ 次修复都失败，停止并质疑架构设计。

---

### verification-before-completion - 完成前验证

**使用场景：** 声称工作完成、已修复或通过时，在提交或创建 PR 之前。

**铁律：** 没有新鲜验证证据就不能声称完成。

**验证模式：**

```
✅ [运行测试命令] [看到: 34/34 通过] "所有测试通过"
❌ "应该通过了" / "看起来正确"
```

**禁止词汇：** should、probably、seems to、"Great!"、"Perfect!"、"Done!"

---

### requesting-code-review - 请求代码审查

**使用场景：** 完成任务、实现主要功能或合并前验证工作是否符合要求。

**核心原则：** 早审查，常审查。

**强制审查时机：**

- 子代理驱动开发中的每个任务后
- 主要功能完成后
- 合并到主分支前

---

### receiving-code-review - 接收代码审查

**使用场景：** 接收代码审查反馈时，在实施建议之前（特别是反馈不清晰或技术上有疑问时）。

**核心原则：** 验证后再实施。提问而非假设。技术正确性优先。

**禁止行为：**

- "你完全正确！"
- "好观点！"
- 不经验证就实施

**应该做：**

- 复述技术需求
- 询问澄清问题
- 有技术理由时提出反驳
- 外部反馈 = 建议而非命令

---

### dispatching-parallel-agents - 分发并行代理

**使用场景：** 面临 2+ 个可并行处理且无共享状态的独立任务。

**核心原则：** 每个独立问题域分配一个代理，让它们并发工作。

**使用场景：**

- 3+ 个测试文件因不同根本原因失败
- 多个子系统独立损坏
- 每个问题可独立理解，无需其他问题上下文

---

### writing-skills - 编写技能

**使用场景：** 创建新技能、编辑现有技能或验证技能部署前是否有效。

**核心原则：** 编写技能就是将 TDD 应用于流程文档。

**TDD 映射：**
| TDD 概念 | 技能创建 |
|-----------|----------|
| 测试用例 | 压力场景 + 子代理 |
| 生产代码 | 技能文档 |
| 测试失败（红） | 无技能时代理违反规则 |
| 测试通过（绿） | 技能存在时代理遵守规则 |

---

### using-superpowers - 使用 Superpowers

**使用场景：** 开始任何对话时，建立如何查找和使用技能的规则。

**核心规则：** 如果有 1% 的可能性某个技能适用，就必须调用该技能进行检查。

**技能优先级：**

1. 流程技能优先（brainstorming、debugging）
2. 实施技能其次

## 工具类技能

### agent-browser - 浏览器自动化

**使用场景：** 需要与网站交互——导航页面、填写表单、点击按钮、截图、提取数据、测试 Web 应用。

**核心命令：**

```bash
agent-browser open <url>           # 打开页面
agent-browser snapshot -i          # 获取可交互元素
agent-browser click @e1            # 点击元素
agent-browser fill @e2 "text"      # 填写表单
agent-browser screenshot           # 截图
agent-browser wait --load networkidle  # 等待加载
```

---

### dogfood - 狗粮测试（探索性测试）

**使用场景：** 系统性地探索和测试 Web 应用，发现 bug、UX 问题。

**产出：** 结构化报告，包含完整复现证据——分步截图、视频和详细复现步骤。

---

### find-skills - 查找技能

**使用场景：** 寻找能做某事的技能——"如何做 X"、"找能做 X 的技能"。

---

### pdf - PDF 处理

**使用场景：** 任何 PDF 相关操作——读取/提取文本表格、合并、拆分、旋转、添加水印、创建、填写表单、加密解密、OCR。

---

### trading-decision - 交易决策

**使用场景：** 分析股票、做出交易决策、评估投资机会、生成 BUY/SELL/HOLD 信号。

**触发词：** 帮我分析股票、选股、股票推荐、交易决策、投资建议。

---

### taro-wechat-miniapp - Taro 微信小程序

**使用场景：** 构建、审查、重构、调试 Taro React 微信小程序。

```
用户请求
    ↓
┌─────────────────────────────────────────────────────────┐
│  using-superpowers (启动检查，调用相关技能)              │
└─────────────────────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────────────────────┐
│  brainstorming (设计阶段)                                │
│  → writing-plans (创建计划)                             │
└─────────────────────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────────────────────┐
│  using-git-worktrees (创建隔离工作空间)                  │
└─────────────────────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────────────────────┐
│  subagent-driven-development (执行计划)                 │
│  ├── 每个任务:                                          │
│  │   ├── test-driven-development (TDD)                  │
│  │   ├── systematic-debugging (调试)                    │
│  │   └── verification-before-completion (验证)          │

```

## 技能类型说明

| 类型         | 说明                    | 示例                               |
| ------------ | ----------------------- | ---------------------------------- |
| **刚性技能** | 必须严格遵循，不能adapt | TDD、debugging、verification       |
| **灵活技能** | 可根据上下文调整原则    | patterns                           |
| **技术技能** | 特定技术领域的操作指南  | agent-browser、taro-wechat-miniapp |
| **参考技能** | API 文档、语法指南      | pdf                                |

---

## 铁律汇总

1. **brainstorming**: 获得设计批准前禁止写代码
2. **test-driven-development**: 没有失败测试就不能写生产代码
3. **systematic-debugging**: 找到根本原因前禁止尝试修复
4. **verification-before-completion**: 没有新鲜证据就不能声称完成
5. **using-superpowers**: 有 1% 可能性适用就必须调用技能
