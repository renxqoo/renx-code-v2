## 核心模块详解

### 1. StatelessAgent (无状态代理)

**位置**: `agent/index.ts`

**职责**:
- 作为 Agent 执行的统一入口
- 组装运行时依赖（LLM、工具执行器、配置）
- 管理生命周期钩子和可观测性
- 保持无状态设计，支持水平扩展

**核心方法**:

```typescript
class StatelessAgent {
  // 主执行入口 - 流式生成
  async *runStream(input: AgentInput, callbacks?: AgentCallbacks): AsyncGenerator<StreamEvent>
  
  // 估算上下文使用量
  estimateContextUsage(messages, tools, contextLimitTokens)
  
  // 获取上下文限制
  getContextLimitTokens(contextLimitTokens?)
}
```

**设计原则**:
- 无状态: 不持有跨运行的持久状态
- 依赖注入: LLM Provider、ToolExecutor 通过构造函数注入
- 可观测性: 通过钩子系统实现，不侵入核心逻辑

### 2. RunLoop (运行循环)

**位置**: `agent/run-loop.ts`

**职责**:
- 控制步骤循环（step loop）
- 决定重试、终止、超时行为
- 协调 LLM 阶段和 Tool 阶段

**核心循环逻辑**:

```typescript
async function* runAgentLoop(runtime: RunLoopRuntime, state: RunLoopState) {
  let stepIndex = 0;
  let retryCount = 0;
  let runOutcome: AgentRunOutcome = 'done';

  while (stepIndex < state.maxSteps) {
    // 1. 检查终止条件（abort、timeout、max retries）
    // 2. 准备消息（可能触发压缩）
    // 3. 执行 LLM 阶段
    // 4. 如果有工具调用，执行 Tool 阶段
    // 5. 否则结束运行
    // 6. 错误处理和重试逻辑
  }
}
```

## 工具系统架构

### Tool V2 架构概览

**设计原则**:
1. 工具规格与执行处理器分离
2. 审批、权限、执行编排在共享基础设施中
3. 会话级和轮次级权限授予是一等公民
4. 工具易于添加，无需修改中心管理器

## 关键设计模式

### 1. 无状态设计

**原则**: Agent 核心不持有跨运行的持久状态

**好处**:
- 支持水平扩展
- 无需会话亲和性
- 故障恢复简单

**实现**:
- `StatelessAgent` 只持有配置和注入的依赖
- 幂等性账本 (`ToolExecutionLedger`) 通过接口注入
- 存储层通过端口 (Ports) 抽象

### 2. 依赖注入

**模式**: 构造函数注入 + 运行时组装

```typescript
// StatelessAgent 构造函数
constructor(
  llmProvider: LLMProvider,      // 注入 LLM 提供者
  toolExecutor: AgentToolExecutor, // 注入工具执行器
  config: AgentConfig            // 注入配置
)

// 运行时组装
const runtime = buildRunLoopRuntime({
  config: { maxRetryCount },
  callbacks: { safe, safeError },
  messages: { prepareForLlmStep, mergeLLMConfig },
  stages: { llm, tools },
  // ...
}, hooks);
```

### 3. 钩子系统

**设计**: 观察者模式，仅用于可观测性

**规则**:
- 钩子不能修改控制流
- 钩子不能覆盖重试/跳过决策
- 钩子失败不影响主流程

**可用钩子**:
- `onRunStart` / `onRunError`
- `onLLMStageStart`
- `onToolStageStart` / `onToolExecutionStart`
- `onRetryScheduled`

### 4. 超时预算

**设计**: 统一的超时管理

```
┌─────────────────────────────────────────────────────────────────┐
│                    超时预算模型                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  总预算 (timeoutBudgetMs)                                       │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                          │   │
│  │  ┌─────────────────────────────────────────────────┐    │   │
│  │  │ LLM 阶段 (llmTimeoutRatio, 默认 0.7)            │    │   │
│  │  │ - 从总预算分配                                   │    │   │
│  │  │ - 创建阶段 AbortScope                           │    │   │
│  │  └─────────────────────────────────────────────────┘    │   │
│  │                                                          │   │
│  │  ┌─────────────────────────────────────────────────┐    │   │
│  │  │ Tool 阶段 (剩余预算)                            │    │   │
│  │  │ - 动态计算剩余时间                               │    │   │
```

## 总结

### 架构优势

1. **清晰的分层**: 应用层、Agent 核心、工具系统、基础设施各司其职
2. **无状态设计**: 支持水平扩展，无会话亲和性需求
3. **可观测性**: 钩子系统提供完整的生命周期观测点
4. **安全性**: 多层权限模型，审批流程，沙箱隔离
5. **可扩展性**: 工具系统易于扩展，无需修改核心代码
